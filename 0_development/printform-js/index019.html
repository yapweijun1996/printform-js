Post-Audit Data Terms (PADDT)

This PADDT-only demo is used to validate behavior when there are no .prowitem rows. The formatter splits PADDT text into .paddt-rowitem segments based on data-paddt-max-words-per-segment. It renders after business footers and uses repeat-footer-logo and repeat-footer-pagenum to place branding and page counters. The text below is intentionally long to trigger segmentation and multi-page output in a single run.

Segmentation policy. PADDT paragraphs are split at word boundaries to reduce reflow risk. Operators can tune paddtMaxWordsPerSegment to stabilize pagination across printers and fonts. Use a consistent font size and line height to reduce drift. This demo keeps the main document empty to expose how header/docinfo/rowheader and footer areas are handled when only PADDT exists.

Page context and numbering. The formatter continues logical page numbers across PADDT pages. If the main document has no row items, the first page still renders the standard header and docinfo, then PADDT starts on the next page. This ensures consistent total page counts and keeps footers visible on PADDT pages when repeat flags are enabled.

Dummy fillers. data-insert-paddt-dummy-row-items controls whether paddt-only pages can use dummy row items; setting it to n keeps audit pages clean. data-insert-dummy-row-while-format-table still allows a single full-height filler if enabled. This demo keeps paddt dummy rows disabled so that only content and footers are visible.

Operational note. Capture screenshots of the first page, first PADDT page, and the final PADDT page to confirm that header/docinfo/rowheader behavior and page numbers align with expectations. The text is long enough to span multiple pages so you can observe the page counter continuity.
